# Flightdeck scripts reference

Reference doc extracted from `SKILL.md`. See [`SKILL.md`](./SKILL.md) for the load-bearing rules; this file holds detailed reference content for on-demand consultation.

## Scripts

```bash
.agents/skills/flightdeck/scripts/<script> [args]
```

**Implementation:** Most scripts are TypeScript under
`skills/flightdeck/lib/flightdeck-core/`. Trampolines under `scripts/`
exec `bun .../src/bin/<script>.ts`; `flightdeck-dashboard` is the Rust dashboard trampoline under `lib/flightdeck-dashboard/`. `bun` remains a hard runtime dependency for the TypeScript scripts.
Functional + integration tests live under `lib/flightdeck-core/tests/`.

| Script | Purpose |
|--------|---------|
| `open-terminal` | Spawn issue worktree(s) with selected harness + optional `--model`/`--effort`. **Never hand-roll issue tmux/terminal commands — use this for issue workflow spawns.** Linear is default; `--tracker github` accepts numeric issues, creates branch `issue-<N>`, fetches `gh issue view`, and emits a self-contained child prompt (no supervisor slash-command recursion). Tmux fallback delegates to `flightdeck-session` in issue mode. Channels flags: `--use-channels` / `--no-channels` toggle the Claude Channels MCP send-path; the env equivalent is `FLIGHTDECK_CLAUDE_CHANNELS=0\|1`. **vstack#216:** `--tracker github --harness claude` defaults to channels-on so the daemon's subscriber binds `cc_transcript` and wake delivery works; explicit `--no-channels` still wins. Channel webhook auth: each spawn generates a per-pane bearer token written to `cc-spawn-<issue>.json` (`channel_token`) and the registry entry (`adapter.cc_channel_token`); `pane-respond` reads it and attaches `Authorization: Bearer <token>` on POSTs. Webhook accepts unauthenticated POSTs only when no token was generated (legacy fallback). |
| `flightdeck-session` | Generic session launcher/attacher. `start` creates a tmux window and registers `.entries[id]`; `attach` records an existing Pi pane by stable pane id. Plan lane uses `start --kind workflow --prompt "Read tmp/brief.md ..."` so per-harness native adapters handle the child launch. The dashboard hook is best-effort here: unsupported dashboard flags or launch failures warn but do not block child pane creation. Both `start` and `attach` finish with `ensure_daemon_for_session` (vstack#213): they consult `flightdeck-daemon health` and respawn the daemon when missing, when master diverged, or when `staleness ∈ {stale-state, stale-inner, pre-active-run, meta-missing}`. `FLIGHTDECK_ENSURE_DAEMON=0` disables the helper; `FLIGHTDECK_DAEMON_BIN` overrides the daemon trampoline. |
| `parallel-groups` | Read/manage parallel issue groups. |
| `flightdeck-state` | Atomic CRUD on the active run's `state.json` under `~/.vstack/flightdeck/projects/<id>/runs/<run-id>/` (`init`/`get`/`set`/`append`/`increment`/`tracked-entries`/`write-entry`/`archive`), activity JSONL sidecar commands (`activity path\|append\|tail\|export`), durable run commands (`run active\|list\|show\|create\|ensure\|terminate\|terminate-active\|import-legacy`), and master-busy lock (`master-busy lock\|unlock\|check`). vstack#227: `statePath()` resolves through the active-run pointer and auto-ensures a run when missing; `flightdeck-state path` returns `<run-dir>/state.json` and `flightdeck-state activity path` returns `<run-dir>/activity.jsonl`. `archive` terminates the active run, writes the durable snapshot under `<run-dir>/snapshots/`, and best-effort stops the per-session `flightdeck-daemon` (vstack#213) so the next `flightdeck-session start` arms a fresh subscriber set. Honors `FLIGHTDECK_DAEMON_BIN` for the daemon trampoline. Exit-code handling: exit 1 (no daemon) is quiet; exit 3 (safety refusal) and any other non-zero status emit a stderr warning identifying the session so an operator can investigate. `FLIGHTDECK_ARCHIVE_SKIP_DAEMON_STOP=1` skips the daemon-stop call entirely. See `workflows/shared/session-watch.md` § 1 for lock semantics. |
| `flightdeck-repo-sync` | Safe post-merge local-main sync helper. `main --project-root <path> --remote origin --branch main --json` validates remote/branch ref components, checks the remote branch with `ls-remote`, fetches with `--no-tags`, `--refmap=`, and an explicit refspec (`+refs/heads/main:refs/remotes/origin/main`) instead of configured fetch/tag refspecs or tag auto-follow, fast-forwards local `main` to `origin/main` only when clean, unambiguous, and free of ignored/untracked collisions with incoming tracked paths, and returns JSON `{status,ahead,behind,dirty_paths,reason,commands_suggested}` plus `diagnostics` for command failures. The collision scan is bounded to incoming tracked paths and existing local candidates; candidate directories are index-aware, so tracked-only dir→file fast-forwards are allowed while directories with ignored/non-tracked entries still block. Large unrelated ignored trees do not block a safe fast-forward. It never resets, stashes, discards, deletes dirty paths, mutates tags, or force-pushes; dirty/ignored-collision/ahead/diverged/missing-branch cases return `blocked`, while true git command failures return `failed` with command/exit/stderr/spawn diagnostics. |
| `flightdeck-daemon` | External wake driver. Polls inner panes, normalizes turn-end events, wakes master with a per-harness payload. Actions: `start \| stop \| status \| health \| events \| ack`. `start` exits `4` for stale `--master` (distinct from usage/missing dependency exit `2`). `stop` exits `0` (stopped or stale pid cleaned), `1` (no daemon for session), or `3` (safety refusal — PID lock missing, flock unavailable, ambiguous state); callers must distinguish 1 (benign no-op) from 3 (warn-and-investigate). Default detached/tmux-window start paths preflight master/inner panes before spawning and fail directly on stderr if invalid. Foreground startup validation failures after daemon cleanup is armed emit `daemon-exited` with `reason:"startup-error"` and `details.error` / `details.exit_code` before exiting. Master respawn trigger: `status --session <S>` says `no daemon` while live entries exist; source panes via `pane-registry list --format inner-panes-live` / `inner-harnesses-live`, re-resolve `$TMUX_PANE` and retry once on exit `4`, and do not yield on unresolved start failure. **vstack#216:** `health --session <S>` includes per-pane `subscriber_status` lines (`bound\|skipped\|stuck\|dead`) read from the daemon-written `fd-daemon-<sessionKey>.subscribers.json` snapshot (heartbeat cadence). Missing snapshot is reported as `subscriber_status=(missing — daemon hasn't written snapshot yet)` rather than silently omitted. Bind-skip throttle / stuck thresholds tunable via `FD_SUB_BIND_SKIP_LOG_INTERVAL_SEC` / `FD_SUB_BIND_SKIP_STUCK_THRESHOLD` (mirrors the pi-specific knobs from #180). **vstack#213:** `flightdeck-session start` / `attach` is the canonical entry point that arms wake delivery and best-effort stops the daemon as part of `flightdeck-state archive`; reconciliation is best-effort safety net, not the primary mechanism for picking up new panes. `health` also exposes `started_at`, `master_pane_id`, `master_harness`, `subscribed_pane_ids`, `subscribed_pane_harnesses` (parallel to subscribed_pane_ids), `state_file_path`, `state_file_inode`, `active_run_id`, and a `staleness: fresh \| stale-state \| stale-inner \| pre-active-run \| meta-missing` enum derived against the live tracked entries (probed via `pane-registry list --format inner-live-json`) with exact-set + per-pane harness comparison; diagnostic lines (`live_inner_probe_error=`, `state_inode_probe_error=`, `active_run_probe_error=`) surface probe failures. Full contract: `workflows/shared/session-watch.md` § 1 / § 6; adapter freshness: `patterns/tmux-monitoring.md`. |
| `flightdeck-dashboard` | Rust TUI dashboard binary. Subcommands include `tui`, `status`, `daemon {start,foreground,stop,status,health,events,ack,tail}`, `launch`, `focus-or-launch`, and `supervise`. The TUI has keyboard/mouse navigation, help/theme/filter/detail/confirm/settings/history popups, dashboard badge/chip legend, and cost/token totals. vstack#227: `tui` reads the active run's `state.json` under `~/.vstack/flightdeck/projects/<id>/runs/<run-id>/` when a matching active run pointer exists; otherwise it shows a no-active landing and History hint instead of auto-loading the newest archive. Use `tui --run-id <id> [--snapshot <timestamp>]` or `tui --archive <path>` for explicit read-only history/legacy views. Write paths are limited to confirmation-gated shells to canonical helpers (`pane-registry remove` for stale entries, `tmux select-window` for focus) plus settings persistence to `~/.vstack/flightdeck/projects/<project-id>/settings.toml`; archive/history views block stale-prune and focus actions. `focus-or-launch` reports path/command/stderr diagnostics when stale tracked dashboard probes or focus/launch verification fail. Lives in `skills/flightdeck/lib/flightdeck-dashboard/`. See `DEVELOPMENT.md` for the build + test workflow. |
| `codex-app-server-spawn` / `-stop` | Idempotent bring-up/teardown of the per-session codex `app-server --listen ws://...` shared by all `codex --remote` panes. |
| `pane-registry` | TrackedEntry↔pane mapping CRUD. `init-entry` writes `.entries[id]`; `init <ISSUE>` is an alias for `init-entry --kind issue`. `find-by-pane` emits `{id,kind}` JSON. `list --format json\|inner-panes\|inner-harnesses\|inner-panes-live\|inner-harnesses-live\|inner-live-json` feeds `pane-poll --batch -` and `flightdeck-daemon start`; use the `*-live` pair for manual daemon respawn. The daemon's max-lifetime handoff uses `inner-live-json` so pane ids and harnesses come from one live snapshot and tmux live-query failures stay non-empty/fail-closed. `refresh-window-names` refreshes `entry.window_name_current` from tmux `#W` and (vstack#214) also recomputes `pane_target` / `window_index` from the live pane snapshot whenever a still-alive `%pane_id` has been moved to new coords by a tmux window renumber; `entry.window` is refreshed only when its stored value is numeric (legacy window-name strings are preserved). Renumbered entries appear in `updated[]`; entries with missing/dead `pane_id` show up in `warnings[]` with reasons `missing-pane-id` / `pane-id-not-live`. `reconcile` invokes the same refresh before its liveness pass so manual reconciles heal stale coords. Domain PR/worktree fields are flattened from `domain.issue`, `domain.github_issue`, or `domain.plan_item` when present. Adapter hydration commands: `hydrate-pi <issue>` (vstack#180) and `hydrate-claude <issue>` (vstack#216) read the matching `*-spawn-<issue>.json` file and write the adapter slots atomically — required when the spawn metadata is created post-registration. `cc-channel-args <issue>` emits `--url ... --transcript ... [--channel-token ...]`; the channel token is included only when the adapter is fresh and a token was generated at spawn time. |
| `pane-poll` | Pane state read. Preferred: `--batch -` from `pane-registry list --format json` (one JSONL object per tracked entry). Emits `window_name_current` from tmux `#W` when available. Passes `kind` to `prompt-classify` so issue-only tags on ad-hoc entries become `domain-mismatch`. Adapter text with a GitHub pull URL on the final non-empty line emits `terminal-state-reached` plus `detected_pr_number` / `detected_pr_url`. Legacy single-pane mode for drift re-polls / manual debug. See `patterns/tmux-monitoring.md` for per-harness adapter routes. |
| `pane-respond` | Send response to a pane. Modes: free-text payload, `--option N`, `--option-multi`, `--keys` (rejected without `--keys-allow-tmux`), `--question <reqID> --answer\|--answer-multi\|--answer-text\|--answers-json\|--reject`. Validates rebase-multi-choice payloads for the preserve/apply/verify triplet. When the caller passes a `%pane_id` (vstack#214), the target is resolved via a live `tmux display-message` lookup first — the registry's cached `pane_target` is only consulted if tmux can't resolve the id — so a renumber-staled cache never misroutes the paste. See `patterns/prompt-handlers.md` for mode selection and `patterns/opencode-questions.md` / `patterns/pi-questions.md` for question routing. |
| `pane-clear-bell` | Atomic chained-command bell clear (no flicker). |
| `pr-conflict-graph` | File-intersection adjacency for a list of PR numbers via `gh pr view --json files`. |
| `label-add` / `label-remove` (in `skills/github/scripts/commands/`) | Add/remove GitHub labels via `gh pr edit` / `gh issue edit`. Emit `pr.labeled` / `pr.unlabeled` / `issue.labeled` / `issue.unlabeled` activity rows when `FLIGHTDECK_MANAGED=1`; silent otherwise. Wrapped by the `github` skill — flightdeck issue workflows call them indirectly. |
| `prompt-classify` | Regex/sentinel + computed-tag matcher that maps pane state to handler tags. `--entry-kind` guards issue-only tags on non-issue entries; omitted kind and `--entry-kind-unknown` fail closed as `domain-mismatch`. `--no-footer-gate` adapter text treats a final-line GitHub pull URL as `terminal-state-reached`. See [`PROMPT-TAGS.md`](./PROMPT-TAGS.md) for the full tag catalog, including `merge-ready-but-unknown`, `pi-bg-task-exit`, `pi-activity-broker`, and `daemon-exited`. |

`pi-bg-task-exit` (vstack#15): the Pi subscriber matches `pi-bridge stream` events of shape `{ type: "event", event: "message_end", data.message.customType: "vstack-background-tasks:event", data.message.details.eventType: "exit" }` and appends a canonical wake row to `WAKE_EVENTS_LOG`:

```
{"ts":"<iso>","pane_id":"%18","harness":"pi","event_type":"bg-task-exit","sequence":17,"task":{"id":"bg-3","status":"failed","exitCode":null,"command":"...","outputBytes":89},"classifier_tag":"pi-bg-task-exit","hash":"<12hex>"}
```

The daemon (`lib/flightdeck-core/src/daemon/loop.ts`) treats the tag as canonical, appends to the per-session events file via `appendEvent`, extends `WAKE_PENDING.in_flight`, and wakes master before emitting structured activity. Non-exit bg-task subscriber signals (for example `details.eventType: "output"`) append activity-only `pi-bg-task-activity` rows with `activity_event_type` and `sequence`; the daemon records them as `bg_task.output_matched` activity without waking master. Master routes terminal rows through `workflows/shared/session-watch.md` § 2 → `workflows/shared/session-handle-prompt.md` § 7; issue mode may then resume `workflows/linear/handle-prompt.md` § 4 for PR/CI/bot-review recovery. The classifier never sees these messages — they are system-role customType messages, not assistant text — so `prompt-classify` has no matching tag and only the daemon path produces them.

`pi-activity-broker`: Pi extensions publish through `globalThis[Symbol.for("vstack.pi.activity")]`; `pi-session-bridge` streams each publication as `{type:"event", event:"vstack_activity", data:{...}}`. The Pi subscriber appends an activity-only `pi-activity-broker` row to `WAKE_EVENTS_LOG`. The TS daemon copies the subset payload into `flightdeck-activity-<session>.jsonl` with the tracked pane id and never wakes master. Set `FLIGHTDECK_PI_ACTIVITY_BROKER=0` to disable this broker drain and rely on legacy custom-message wake paths only.

`pi-rate-limit-*`: the Pi subscriber runs every non-custom `message_end` through the rate-limit decider, piping event JSON on stdin (never argv). Classifier rejections append activity-only `pi-rate-limit-skipped` rows with `event_type:"rate_limit_skipped"` and `reason:"non-assistant"|"no-stopreason"|"stopreason-mismatch"|"no-prose"`; non-assistant rows stop there and never enter prompt classification. Positive decisions append activity-only `pi-rate-limit-retry` rows (`agent.rate_limit_retry`, attempt + `next_retry_at`) and schedule the steer; a later healthy assistant rejection resets the subscriber budget and appends `pi-rate-limit-resolved` (`agent.rate_limit_resolved`). Exhaustion appends `pi-rate-limit-exhausted` (`agent.rate_limit_exhausted`) as an activity/advisory signal only, then the subscriber continues; the exhausted row itself does not wake master and does not fall through to normal completion/blocking. Any later completion or blocked handling comes from independent events or daemon polls. Decider missing/nonzero/malformed output appends `pi-rate-limit-decider-error` (`daemon.warning`) and logs the stderr/output tail.

Pi subscriber binding guard: before spawning a Pi subscriber from registry metadata, the daemon verifies that `/proc/<pi_pid>/cwd` resolves inside the tracked entry `cwd` and that the bridge session id matches the entry's recorded `adapter.pi_session_id`. If stored metadata points at a sibling Pi process, the daemon re-scans `pi-bridge list --json` for the matching `(cwd, sessionId)` tuple; if no match is available, it skips the spawn so the next reconcile tick can retry. Before initial question drain, the subscriber probes `pi-bridge state` through `timeout ${FD_ADAPTER_READ_TIMEOUT_SEC}s`; timeout or bridge error logs `pi-sub-session-preflight-error`, malformed/missing-session output logs `pi-sub-session-preflight-malformed`, and both fail open to stream attach. On stream connect, the subscriber records `pi_session_id` from `bridge_hello.state`; a mismatch with the expected session id emits `subscriber.mismatch` activity, quarantines any same-drain rows for that pane, reaps the old subscriber process group, and retries replacement spawn without reattaching stale pidfiles until binding succeeds. Force-respawn resolves the harness from the live registry entry before falling back to cached/default harnesses, so mixed-harness sessions keep Pi subscribers attached to Pi panes even after reap clears subscriber state. Reap signals target only the captured process group; the daemon does not fall back to a positive PID without identity proof. Deferred reap cleanup removes pid/log files only when the pidfile still points at the reaped pid, preserving replacement subscriber files after force-respawn.

Activity sidecar: `flightdeck-state init` records `activity_path` as the active run's `<run-dir>/activity.jsonl` (vstack#227); the legacy project-local `<project>/tmp/flightdeck-activity-<TMUX_SESSION>.jsonl` is no longer the live target. `flightdeck-state activity path|append|tail|export` exposes the path, appends a normalized event, tails recent rows, or exports JSONL/Markdown. `activity export` accepts `--session <name>` and `--state-file <path>` for parity with `path` / `tail` / `append`, so dashboards and post-mortems can resolve sidecars without an active tmux session. Retention caps are 5,000 events and 10 MiB per live sidecar; oversized `details` are truncated to a 16 KiB budget. Daemon emission is curated: daemon/subscriber lifecycle, wake-delivery failures, subagent completions, bg-task exit/output rows, questions, and Pi broker rows. Workflow/github/linear helper emission is gated by `FLIGHTDECK_MANAGED=1 || FLIGHTDECK_ACTIVITY_FILE` so standalone wrapper use stays silent. Post-merge repository sync emits `repo.main_synced`, `repo.main_sync_blocked`, or `repo.main_sync_failed` with `ahead`, `behind`, `dirty_paths`, `reason`, and `commands_suggested` in `details`.

Durable run commands: `flightdeck-state run create --project-root <path> --tmux-session <name> [--state-dir <dir>]` initializes `~/.vstack/flightdeck/projects/<project-id>/runs/<run-id>/` with `metadata.json`, `state.json`, `activity.jsonl`, and sets `active-runs/<tmux-session>.json` plus legacy `active-run.json`. vstack#227: the run-store root is overridable via `FLIGHTDECK_RUN_STORE_ROOT` (tests/isolation only); `--state-dir` / `FLIGHTDECK_STATE_DIR` are only consulted by the legacy migration shim. `run ensure --tmux-session <name>` reuses only that tmux session's active run, creates one when absent, creates fresh after termination, and finalizes stale runs only after tmux liveness succeeds and every recorded pane id is absent. Other tmux sessions' active runs do not block it. Missing same-session metadata, same-session pointer mismatch, or failed tmux liveness checks fail closed with recovery guidance. CLI helpers (`get`/`set`/`tracked-entries`/`activity`/etc.) call into the run-store with `checkStale: false` so an idle helper invocation does not surprise-rotate a run when no tmux session is up; only `run ensure` triggers the liveness check. `run active --tmux-session <name>` returns one session pointer; `run active --all --json` returns all active pointers. `run show`, `run terminate`, `run terminate-active`, and `run import-legacy` emit JSON; `run list --json` emits machine output while plain `run list` prints tab-separated rows. `run terminate` writes a final state snapshot under `<run-dir>/snapshots/<TS>.json` (paired with `<TS>.activity.jsonl`), copies `summary_path` / `--summary-path` to run `summary.md` when available, marks `<project>/tmp/flightdeck-state-<S>.json` and its sidecar as `.migrated` if present, and clears only the active pointer for the terminated run. `run terminate-active --tmux-session <name>` targets that session. `flightdeck-state archive` is a thin wrapper around `run terminate-active` that prepends the `session.completed` activity append and the daemon-stop call. If start/attach creates a fresh active run and then aborts before `pane-registry init-entry`, the launcher terminates only that newly-created run; reused active runs are left intact. Dashboard self-launch still sets `FLIGHTDECK_SKIP_ACTIVE_RUN=1`, but CLI helpers auto-create a run as soon as the dashboard's pane-registry writes touch `flightdeck-state set`, so the dashboard entry ends up in the same active run as the user's linear/github entry. `run import-legacy [--state-dir <dir>]` copies `flightdeck-state-*.json.archive` plus matching activity archives into deterministic imported run ids and leaves legacy archives untouched.

`daemon-exited`: the daemon emits this lifecycle row during cleanup when it exits for `master-gone`, `signal-term`, `signal-int`, `startup-error`, or another recorded reason. It writes directly to the per-session `EVENTS_FILE` under `SESSION_LOCK` (not `WAKE_EVENTS_LOG`), with `pane_id` set to the master pane id so pane-keyed drains include it. Foreground startup validation failures emitted as `startup-error` include diagnostic fields under `details` (for example `error` and `exit_code`) so a drained event is actionable without grepping the daemon log; ordinary runtime lifecycle exits such as `master-gone` and `session-gone` carry only the lifecycle fields (`reason`, `pid`, `master_id`) unless a caller records extra details:

```
{"ts":"<iso>","pane_id":"%25","event_type":"daemon-exited","reason":"master-gone","master_id":"%25","pid":12345,"hash":"<12hex>","tag":"daemon-exited","stable_age_sec":0,"details":{"event_type":"daemon-exited","reason":"master-gone","master_id":"%25","pid":12345}}
```

`session-watch.md` routes `daemon-exited` as a daemon-lifecycle signal, not a pane-prompt classification. It records the reason and follows the master respawn flow in `workflows/shared/session-watch.md` § 1 / § 6 before yielding.
